# Snell Credential Contract

部署、迁移、修复或客户端验收前，先唯一确定 PSK 的来源。允许的来源只有：

- 用户在当前任务中明确提供；
- 当前任务为这个目标新生成；
- 用户授权从明确命名的目标 VPS 配置路径或本机 Surge profile 读取。

禁止搜索隐藏凭据缓存、密码管理器、无关 profile 或 shell 历史；不跨 VPS、账号或环境复制 PSK，也不借用其他节点的值。读取某个凭据不等于获准把它用于另一个服务端或客户端。

新 PSK 使用独立高熵值，只写入目标服务端配置和目标客户端 profile。通过权限为 `0600` 的暂存文件、标准输入或目标应用的受保护配置通道交付；不要把 PSK 放进命令行参数、进程列表、journal、审计 JSON、host 清单、聊天或最终回复。写入时关闭 shell xtrace，不打印配置全文。

从 `psk = <value>` 配置行读取既有 PSK 时，只移除第一个键和值分隔符及其周围空白，保留其余完整原值，包括末尾的 `=` padding。不能按所有 `=` 分列再取第二段；这种写法会静默截断 Base64 PSK，而 Surge 的 TCP 延迟探针仍可能返回数值，必须同时要求服务端没有对应的 `Decryption failed`，且 UDP relay、出口 IP 或目标业务请求通过。

服务端与客户端都需要同一 PSK 时，先在一个已授权来源生成或读取，再分别写入两个明确目标；不要在中间输出明文。无法建立安全交付路径时停止部署，并向用户索取可授权的来源或目标配置入口。

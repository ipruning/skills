# Audit Protocol

角色定义各档位通用。Sub-agent 协议一节只在开独立 sub-agent 时生效。

## Sub-agent 协议

别让 sub-agent 看到你自己的倾向。每个只给它需求描述、目标用户、预期使用场景、市场和语言背景、以及三到五个相关搜索关键词。让它们尽量去搜或查真实证据，而不是只靠模型先验。

每个 sub-agent 必须输出：

- 一到两句话的结论。
- 三条最强理由。
- 两个一旦不成立就会翻转结论的关键假设。
- 一条最可能推翻它自己观点的证据。
- 置信度：high、medium 或 low。
- 证据级别，标结论主要依据的那一级：public fact、user-provided fact、role simulation 或 model prior。

## 五个角色

- **Bull，需求存在方**：在 Dream Case 的理想形态假设下找需求为真的最强论证。指出 first-principle 需求、现有 workaround、被容忍的痛、付费信号，以及最可能先跑通的首批用户和场景。别停在「更方便」，要说清行为为什么会改变。
- **Bear，假需求猎手**：攻击二阶优化叙事，「更快」「更方便」「更智能」「更实时」「全自动」。找摩擦、切换成本、替代品，找出把需求判成宽而浅或只在窄场景成立的证据。追问用户是否真需要这个程度的方案。
- **User Reality，具体场景**：模拟三到五个具体用户类型。每个说清人设是谁、与需求相关的关键属性，当前 workaround 的确切步骤和时间成本，什么时刻触发使用，什么时刻变得烦人或无关，一到两个流失原因，每周用几次，以及在一天里算工具、娱乐、打扰还是根本不在。
- **Validation Design，现实验证**：列出 AI 无法知道的事。设计最小验证动作，访谈、concierge test、付费测试、落地页、原型演示或人工代办。设计问题时排除诱导性提问和样本偏差。客气话和兴趣不计为采用证据。
- **Source Feasibility，交付风险**：审供给侧。数据从哪来，是否稳定、结构化、新鲜、准确、合规、可防守、负担得起、容易被封。找出最小可用覆盖和最小可用新鲜度。

## 合成顺序

拿到 sub-agent 输出后按这个顺序合成：先找多个角色一致的共识信号，再保留分歧、指出结论在哪个假设上分叉并判断哪个假设更可能，然后把 public fact 和 user-provided fact 与 role simulation、model prior 分开，再回答 Dream Case 下 JTBD 是否存在、方案是否是好方案、差异化是否有价值，最后落到决策信号，信号为 `continue` 时定义最小的 wedge 和现在不要建的东西。

## 访谈问题库

别问假设性问题，它们把判断外包给了用户：

- 「你会用 X 吗？」
- 「你觉得这个想法怎么样？」
- 「你会为它付钱吗？」
- 「如果有这个东西，你会感兴趣吗？」

要问行为回溯问题，从用户实际做过的事反推需求：

- 「上次这事发生是什么时候？」
- 「你当时怎么解决的？」
- 「多久发生一次？」
- 「不解决会怎样？」
- 「你试过哪些替代方案？」
- 「为解决它你花了什么：时间、钱、精力、关系、注意力？」
- 「什么会让你宁可保持旧方式也不切换？」
- 「最近一次因为延迟、信息缺失或流程摩擦造成的具体损失是什么？」

## 高危信号

出现这些时提高怀疑：

- 谈概念不谈具体场景。
- 从 power user 外推到大众市场。
- 把技术可行性当成需求证明。
- 把可见的第三方平台或 API 数据当成稳定的数据资产。
- 把新鲜感误当留存。
